# План реализации

## Результат первой версии

Пользователь открывает статический сайт, выбирает профессию, дату рождения и дату прогноза, видит свой знак зодиака и получает один развлекательный IT-прогноз от локальной LLM. В настройках можно выбрать провайдера и модель, при этом Ollama остаётся обязательным и рабочим вариантом.

## Этапы

### 1. Старт проекта

- Инициализировать Vite-проект с React и TypeScript.
- Включить строгий TypeScript, настроить Vitest и базовые команды `dev`, `test`, `build`.
- Создать `.gitignore`, `.env.example` без секретов и папку `docs/`.
- Добавить пустой каркас модулей из архитектурного документа.

**Готово, когда:** чистый проект запускается, тесты и сборка проходят.

### 2. Доменная логика и форма

- Составить локальный список IT-профессий: например, frontend-разработчик, backend-разработчик, QA-инженер, DevOps-инженер, аналитик данных, UX/UI-дизайнер, product manager, системный администратор.
- Реализовать чистую функцию определения знака зодиака по месяцу и дню рождения.
- Реализовать валидацию: профессия выбрана, даты валидны, дата прогноза выбрана.
- Собрать форму без подключения LLM и показать пользователю вычисленный знак.

**Готово, когда:** граничные даты каждого знака покрыты unit-тестами, а форма не позволяет начать генерацию с неполными данными.

### 3. Генерация и локальный Ollama

- Описать общий интерфейс AI-провайдера и типы запроса/ответа.
- Реализовать `OllamaProvider` с настройками URL и имени модели.
- Собрать управляемый промпт из профессии, знака и даты прогноза.
- Добавить состояния загрузки, успеха и понятных ошибок подключения/отсутствующей модели.
- Добавить кнопку повтора, доступную только после завершения прошлого запроса.

**Готово, когда:** при запущенном Ollama и установленной модели можно получить прогноз из браузера; при выключенном Ollama пользователь видит инструкцию, а приложение не ломается.

### 4. Настройки и переключаемость поставщика

- Добавить экран или блок настроек: активный провайдер, base URL, модель.
- Сохранять только безопасные локальные настройки в `localStorage`.
- Добавить заготовку `CloudProvider` за тем же интерфейсом, но не требовать ключ или сервер для учебной версии.
- Для режима с пользовательским ключом хранить его только в памяти вкладки и явно предупреждать, что это демонстрационный вариант.

**Готово, когда:** смена модели Ollama не требует правок формы или результата; типы и тесты подтверждают единый контракт провайдеров.

### 5. Качество и публикация

- Реализовать мобильную и десктопную вёрстку по `docs/design.md`.
- Проверить доступность формы с клавиатуры, тексты ошибок и состояние загрузки.
- Прогнать сценарии с несколькими профессиями, знаками и датами.
- Опубликовать статическую сборку на выбранном хостинге и приложить инструкцию запуска Ollama для демонстрации.

**Готово, когда:** `build` проходит, ключевые сценарии протестированы, ссылка на статический сайт и инструкция локального запуска доступны проверяющему.

## Приоритеты

Сначала необходимо довести до рабочего состояния путь «форма → Ollama → прогноз». Смена модели и визуальная полировка не должны задерживать его. Облачный провайдер — архитектурная готовность, а не обязательная функция первой демонстрации.

## Риски и решения

| Риск | Решение |
| --- | --- |
| Ollama не запущен или модель не установлена | Ясное сообщение с действиями пользователя; не делать молчаливый fallback в облако. |
| CORS в браузере | Проверить origin локального dev-сервера и показать настройки `OLLAMA_ORIGINS` при необходимости. |
| Локальная модель пишет слишком общий текст | Уточнить системную инструкцию, добавить профессиональные ситуации в справочник и несколько примеров поведения. |
| Секрет облачного API попадает в браузер | Не хранить ключ в `.env` сборки; для публичного облачного режима позже использовать BFF. |
| Дата рождения воспринимается как чувствительная | Не отправлять её целиком в LLM: передавать только вычисленный знак. |

## Что сознательно отложено

- Реализация дизайн-системы в коде.
- Авторизация, сервер, база данных и история прогнозов.
- Мультиязычность.
- Стриминг ответа, рейтинги прогнозов и социальные функции.
- Публичный облачный режим с серверным хранением API-ключа.
